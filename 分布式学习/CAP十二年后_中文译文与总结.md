# 《CAP 十二年后：规则如何改变》

Eric Brewer，IEEE *Computer*，2012 年 2 月

> 译注：本文中的 CAP 一致性指“单副本一致性”（single-copy consistency），并不等同于 ACID 中“满足数据库约束”的 Consistency。文末保留英文参考文献，以便查阅原始来源。

## 摘要

CAP 定理断言：任何通过网络共享数据的系统，至多能同时拥有三种理想性质中的两种。不过，只要明确地处理网络分区，设计者仍可在一致性与可用性之间进行优化，因而在三者上获得适合应用场景的折中。

## “三选二”为何具有误导性

CAP 定理提出后的十年间，设计者和研究者用它（有时也误用它）为各种新型分布式系统背书；NoSQL 运动也把它作为反对传统数据库的理由。CAP 定理说，任何网络化共享数据系统最多具有下列三项理想性质中的两项：

- **一致性（C）**：等价于系统只有一份最新的数据副本；
- **高可用性（A）**：数据（包括更新）始终可用；
- **网络分区容错性（P）**：能够容忍网络分区。

“三选二”的说法完成了它原本的任务：让设计者看到更大的系统设计空间。过去十年由此出现了丰富的新系统，也引发了关于一致性和可用性相对价值的讨论。但这种说法一直过度简化了三者的张力。现在应当认识到：CAP 实际禁止的设计空间极小——仅仅禁止在发生分区时同时做到**完美可用**和**完美一致**，而网络分区本来就很少见。

在分区发生时，设计者仍必须在 C 与 A 之间选择；但如何处理分区及如何从中恢复，存在极大的灵活性。现代 CAP 的目标应是：针对具体应用，最大化那些合理的一致性与可用性组合。这样的思路既包含分区期间的运行计划，也包含分区后的恢复计划，因而超越了 CAP 历来被认为的限制。

理解 CAP 最容易的方式，是设想两个节点被分区隔开。若允许至少一侧更新状态，双方状态便可能不一致，因而失去 C；若要保持一致，则至少一侧必须表现为不可用，因而失去 A。只有节点能够通信时，才可能同时保留 C 和 A，这时相当于放弃 P。对于广域系统，人们通常认为 P 无法放弃，因此似乎只能艰难地在 C 与 A 之间抉择。某种意义上，NoSQL 强调优先可用、其次一致；遵循 ACID 的数据库则反过来。

事实上，CAP 正是源自这样的实际讨论。20 世纪 90 年代中期，作者与同事构建了多种基于集群的广域系统（可视为早期云计算），包括搜索引擎、代理缓存和内容分发系统。由于营收目标和服务合同都高度重视可用性，他们经常选择通过缓存、记录更新以便稍后协调等策略优化可用性；代价是降低一致性。

这种“C 与 A”的讨论最先以 ACID 对比 BASE 的形式出现，但当时反响不佳：人们喜欢 ACID，不愿放弃它。CAP 的目的正是为探索更宽广的设计空间提供理由，因此采用了“三选二”的表述。定理在 1998 年秋首次提出，1999 年发表，并在 2000 年 PODC 的主题演讲中促成了后续证明。

“三选二”有几层误导：第一，分区很少发生，系统未分区时没有理由牺牲 C 或 A。第二，C/A 的选择可以在一个系统内以很细的粒度多次发生：不同子系统、不同操作、甚至不同数据或用户都可作不同选择。第三，三种性质都不是二元的。可用性显然从 0% 到 100% 连续变化；一致性也有多个等级；连“分区”也有细微差异，例如系统内不同节点可能并不一致地认为分区存在。

因此，关键是超越传统的、被动应对分区的方式。因为分区罕见，CAP 应允许系统在大多数时间同时拥有完美 C 和 A；但当分区存在或被认为存在时，系统应检测它、进入显式的分区模式（可限制部分操作），并在通信恢复后启动恢复流程，恢复一致性并补偿分区期间的错误。

## ACID、BASE 与 CAP

ACID 与 BASE 是一致性—可用性光谱两端的设计哲学。ACID 强调一致性，是传统数据库的路线；BASE 是为概括当时新兴的高可用设计而提出的缩写：**基本可用（Basically Available）、软状态（Soft state）、最终一致（Eventually consistent）**。软状态和最终一致性在分区下表现良好，因此促进可用性。现代大规模广域系统（包括云）通常混合使用两者。

ACID 和 CAP 的关系更复杂，也常被误解：两者中相同字母并不代表相同概念；而选择可用性也只会影响 ACID 的部分保证。

- **原子性（ACID-A）**：即便优先可用，分区两侧仍应使用原子操作；更高层的原子操作还会简化恢复。
- **一致性（ACID-C）**：ACID 的 C 指事务保持唯一键等所有数据库规则；CAP 的 C 仅指单副本一致性，是前者的严格子集。分区期间无法跨分区维持 ACID 一致性，恢复阶段必须修复它。若要在分区中保持不变量，往往不可能，因而必须明确哪些操作应被禁止，以及如何恢复不变量。
- **隔离性（ACID-I）**：这是 CAP 的核心。若系统要求 ACID 隔离性，分区时至多一侧能运行；可串行化一般需要通信，所以无法跨分区成立。较弱正确性定义可以依靠恢复期补偿而跨分区实现。
- **持久性（ACID-D）**：没有理由为可用性放弃持久性，尽管开发者可能因成本采用软状态。分区恢复时，可能撤销曾经持久化、却在当时未知地违反不变量的操作；拿到双方的持久历史后，系统即可检测并纠正它们。分区两侧各自执行 ACID 事务，会让恢复更容易，并为补偿事务提供框架。

## CAP 与延迟的联系

经典 CAP 解读忽略延迟，但实践中延迟和分区紧密相关。CAP 的本质通常发生在一次**超时**时：程序必须作出分区决策：

1. 取消操作，从而降低可用性；或
2. 继续操作，从而冒不一致风险。

通过 Paxos 或两阶段提交反复重试通信，只是在延后这个决定；无限期重试，实质上就是选择 C 而非 A。务实地说，分区就是通信的时间上界：在该上界内不能取得一致，便意味着对该操作而言发生了分区，必须在 C 与 A 之间选择。设计问题的核心是：通信中断时，两侧是否还要继续向前推进？

这种观点有三项后果。其一，不存在全局统一的“分区”：有些节点检测到分区，另一些未必。其二，节点可检测分区并进入分区模式，这是优化 C 与 A 的中心手段。其三，设计者可以依据目标响应时间有意设定时间界；界限越紧，系统越可能进入分区模式，甚至把单纯的网络变慢误判为分区。

有时，放弃强一致性是为了避免跨广域维持一致的高延迟。Yahoo 的 PNUTS 异步维护远端副本，容许不一致，但使主副本靠近用户，从而降低延迟；单个用户的数据天然按其通常所在地分布。Facebook 则让主副本固定在一处：远程用户通常访问更近但可能过期的副本；用户更新页面后，更新会直接发往主副本，该用户的读请求也会在一小段时间内直达主副本，虽然延迟更高。约 20 秒后，流量再回到近端副本，此时它应已反映更新。

## 管理网络分区

面对分区，设计者最困难的任务是减轻其对一致性和可用性的影响。关键是**显式管理分区**：不仅要检测，还要设计恢复流程，并为分区期间可能被破坏的每条不变量制订计划。流程分三步：

1. 检测分区开始；
2. 进入显式分区模式，必要时限制部分操作；
3. 通信恢复后启动分区恢复。

最后一步旨在恢复一致性，并补偿程序在分区期做出的错误。

正常运行是一系列原子操作，因此分区总是发生在两个操作之间。超时后，检测到的一侧进入分区模式；若确实发生分区，双方最终都会进入该模式，但也可能仅一侧感知到分区。另一侧若仍能完成所需通信，或操作根本无需通信，结果仍保持一致；但已检测一侧可能已执行不一致操作，所以必须进入分区模式。使用法定人数（quorum）的系统便是例子：拥有法定人数的一侧能继续，另一侧不能。支持离线操作的系统，以及 JGroups 等原子多播系统，也都具有分区模式。

进入分区模式后有两种策略：要么限制某些操作以降低可用性；要么记录额外操作信息，为恢复提供依据。继续尝试通信能够帮助系统判断分区何时结束。

### 哪些操作应继续？

这主要由必须维护的不变量决定。给定一组不变量，设计者要判断：某项不变量是否必须在分区模式中维持，还是允许暂时违反、待恢复时修复。例如，若不变量是表中键唯一，通常可冒险允许分区两侧产生重复键；恢复时容易检测，只要能合并便容易恢复。

若某不变量在分区期间必须保持，就要禁止或修改可能违反它的操作。一般无法知道操作是否真的会违反，因为另一侧状态不可知。信用卡扣款等外部化事件便属此类：应记录意图，等恢复后再执行。它们通常属于有明确订单状态的大工作流，延迟执行的负担较小；此时以用户不易察觉的方式牺牲了 A——用户只知道下了单，系统稍后会完成。

这也带来用户界面挑战：系统要表达“任务正在进行，但尚未完成”。离线操作本质上是长分区，已有较多研究。例如 Bayou 日历用不同颜色显示可能不一致的暂定条目。工作流中的邮件通知、Google Docs 等有离线模式的云服务，也常使用类似提示。

之所以要围绕显式的高层原子操作（而非只谈读写）设计，是因为这样更容易分析每个操作对不变量的影响。设计者应建立“操作 × 不变量”的矩阵：对每一格判断该操作是否可能破坏该不变量；若可能，就决定禁止、延迟还是改写该操作。实际决策也可取决于已知状态和参数。例如某些数据有归属节点时，操作通常可在归属节点继续，而不应在其他节点继续。

追踪两侧操作历史的最佳方式是**版本向量**，它记录操作间的因果依赖。每个元素是 `(节点，逻辑时间)`，对对象的每个更新节点保存其最后更新时间。给定对象版本 A、B：若所有共同节点上 A 的时间都大于等于 B，且至少一个更大，则 A 比 B 新；若无法排序，说明更新是并发的，因而可能不一致。拥有双方版本向量历史后，系统就能区分已知有序的操作和并发执行的操作。若选择优先可用，因果一致性在一般情形下是可达到的最好结果。

## 分区恢复

通信最终恢复，分区结束。分区期双方都可用并不断向前推进，但某些操作被延后，一些不变量也可能被违反。由于分区模式中保存了仔细的日志，系统此时知道双方的状态和历史。状态本身不如历史重要：历史能让系统推断哪些操作实际破坏了不变量，以及已经对用户输出了哪些结果。恢复要解决两件难题：

1. 使双方状态重新一致；
2. 补偿分区模式中造成的错误。

修复当前状态通常较容易：从分区发生时的状态出发，以某种方式向前重放双方操作，并始终维护一致状态。Bayou 显式地将数据库回滚到正确时点，再以确定性的顺序重放完整操作集合，使所有节点到达同一状态。CVS 等源码控制系统也从共同的一致点出发，向前应用更新以合并分支。

并非所有冲突都能自动合并。CVS 有时需要用户手工处理冲突；具备离线模式的 Wiki 也常在最终文档中保留需要人工编辑的冲突。反过来，有些系统可通过限制操作种类而始终合并冲突。Google Docs 的文本编辑只允许应用样式、插入或删除文本；虽然一般的冲突解决不可判定，但实践中可以在分区期间约束操作，以确保恢复时自动合并。延迟高风险操作是实现这种策略的简单途径。

使用**可交换操作**是自动状态收敛最接近通用框架的方法：系统拼接日志、排序后执行；可交换性意味着可以把操作重排为首选的一致全局顺序。但仅使用可交换操作并不容易：加法可交换，带上下界检查的加法却不可交换，例如余额为零时的扣减。

INRIA 的 Marc Shapiro 团队提出的**无冲突复制数据类型（CRDT）**显著推进了状态收敛。CRDT 可证明在分区后收敛，做法是：要么确保分区期间的所有操作可交换；要么将值表示成格（lattice），确保操作相对该格单调递增。后一方法把状态收敛到双方值的最大值。Amazon 购物车就近似如此：分区后两侧购物车取并集；代价是已删除的商品可能重新出现。

CRDT 也可实现既添加又删除元素的分区容错集合：分别保存“已添加”和“已删除”的两个集合，集合成员资格为两者之差。两个简化集合均收敛，所以差集也收敛；系统未分区时可清理同时出现在两集合中的删除项。清理通常只能在非分区时进行。也就是说，分区中仍要禁止或推迟某些操作，但它们只是清理，不会限制用户感知到的可用性。通过 CRDT，设计者可选择 A，同时仍保证分区后状态自动收敛。

## 补偿分区期间的错误

除了计算分区后的状态，更难的是修复分区期间造成的错误。对分区模式操作进行跟踪和限制，能明确哪些不变量可能被违反，从而让设计者为每个不变量准备恢复策略。系统通常在恢复时发现违规，再实施修复。

修复方式包括：简单的“最后写入者胜出”（会丢弃部分更新）、更智能的操作合并，以及人工升级处理。航班超售就是人工处理的例子：登机相当于分区恢复，不变量是座位数不少于乘客数。乘客过多时，总有人失去座位，客服应以某种方式补偿。这里还存在已外部化的错误：若航空公司从未向乘客承诺有座位，修复就容易得多。因此应延迟风险操作；恢复时事实已清楚。补偿的核心是设计既能恢复不变量、又能更广泛纠正外部化错误的补偿操作。

CRDT 在技术上只支持本地可验证的不变量，这使它不需补偿，但也降低了表达能力。实际方案可以用 CRDT 完成状态收敛，同时暂时允许全局不变量被破坏，待分区结束、状态收敛后再执行必要补偿。

修复外部化错误通常需要保存外部输出的历史。作者以醉酒后打电话为比喻：人清醒后的状态可能无碍，但通话日志仍记录了一些可能是错误的电话；若本人不记得这些电话，就难以补偿它们带来的麻烦。机器系统亦然：分区时可能重复执行订单。若系统能区分两笔有意订单和一笔订单被重复执行，便可取消重复项；若已经对外产生效果，可自动发邮件承认错误、说明已修复，并附上折扣券。没有适当历史时，发现错误的负担便落到客户身上。

补偿事务也被用于处理长事务。长事务面对的是分区决策的一个变体：长时间持锁以确保一致，还是尽早释放锁、让其他事务看到未提交数据以提高并发？例如将所有员工记录作为一个事务更新，普通串行化会锁住全部记录。补偿事务把大事务拆为一个 **Saga**：由多个会逐步提交的子事务组成。若要中止大事务，系统必须为每个已经提交的子事务发起新的事务来纠正其影响，这就是补偿事务。

目标通常是避免中止那些已经使用错误提交数据的其他事务（即避免级联中止）。正确性不再取决于可串行化或隔离性，而取决于事务序列对状态与输出的**净效果**：补偿后，数据库是否本质上等价于那些子事务从未执行时应有的状态？该等价还必须计入外部动作。例如，为重复购买退款显然不同于从未扣款，但可被视为足够的等价。分区恢复也同样如此：服务或产品提供者无法总是直接撤销错误，但应承认错误并采取新的补偿行动。如何将这些思想最好地用于分区恢复，仍是开放问题。

### ATM 的补偿问题

自动取款机（ATM）似乎应选强一致性，但实践中 A 往往压过 C：更高可用性意味着更高收入。其操作包括存款、取款和查询余额；关键不变量是余额应大于等于零。只有取款可能破坏它，另两类操作可以始终执行。

分区时，设计者可因不知道真实余额而禁止取款，但这会损失可用性。现代 ATM 往往进入备用（即分区）模式，将净取款额限制为最多 `k`（如 200 美元）。在限额内取款正常，达到限额便拒绝，从而允许部分取款并限制风险。

分区结束后，必须恢复一致性并补偿错误。由于操作可交换，状态恢复很容易；补偿有多种形式。最终余额为负违反不变量，ATM 已经吐钞，错误已外部化，银行可收取透支费并要求还款。风险有上限，问题通常不严重。但也可能在分区中某时余额为负（ATM 不知情），后续存款又使余额转正；银行可追溯收取透支费，也可因客户已补足资金而忽略这次违规。总体而言，受通信延迟影响，银行系统并非依赖强一致性保证正确，而是依赖审计与补偿；“支票套现”也是例子——客户在网点尚未通信前多处取款，事后才被发现并可能受到法律追偿。

## 结论

系统设计者不应在分区存在时盲目牺牲一致性或可用性。通过仔细管理分区中的不变量，可以同时优化两者。版本向量和 CRDT 等技术正在进入更易用的框架，这类优化会更普及；但与 ACID 事务相比，它要求更深入地理解具体服务的不变量和操作，最佳方案也必然依赖具体业务。

## 工程总结

1. **CAP 不是静态的“三选二”。** 未分区时应尽可能同时实现高可用和强一致；真正的取舍只在超时/分区窗口内发生，而且可细化到数据项、操作和用户。
2. **把超时当作业务决策点。** 超时阈值既决定用户等待时间，也决定系统多频繁进入“分区模式”；不要把它仅当作网络参数。
3. **显式设计三阶段状态机。** 正常运行 → 分区模式 → 恢复。分区模式要规定哪些操作继续、受限、延迟或只记录意图；恢复阶段要合并状态并执行补偿。
4. **从不变量而非数据库品牌出发。** 为每个“操作 × 不变量”组合明确策略。可暂时违反且可修复的不变量可优先保 A；不可逆的外部副作用应延后、限制额度，或交由人工审批。
5. **日志和因果信息是恢复的前提。** 仅存最终状态不够；应保存操作历史、因果关系（如版本向量）以及对用户和外部系统已产生的输出。
6. **尽量让操作可合并。** 使用可交换操作、CRDT、归属节点/单主策略，或限制分区期的操作集合，可大幅降低人工冲突处理和补偿成本。
7. **补偿是产品能力，不只是存储能力。** 退款、取消订单、客服通知、折扣券、审计等都属于分区恢复的一部分；UI 必须表达“已受理但未完成”或“暂定”的状态。
